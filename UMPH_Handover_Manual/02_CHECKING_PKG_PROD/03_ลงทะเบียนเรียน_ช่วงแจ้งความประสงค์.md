
# คู่มือ API: ระบบแจ้งความประสงค์ลงทะเบียนเรียน (`MIS_API_REG_ENROLL_PRE_ENROLL`)

## 📌 1. Overview ของ Package

Package นี้ถูกเรียกใช้งาน (Call) จากฝั่ง Application/Web เพื่อจัดการกระบวนการ **"แจ้งความประสงค์ลงทะเบียนเรียน"** ของนักศึกษา โดยมีหน้าที่ตั้งแต่การดึงรายวิชามาแสดงผล, การตรวจสอบเงื่อนไข (ตารางสอบชน, หน่วยกิตเกิน), การบันทึก/ลบ/แก้ไข คำร้อง, ไปจนถึงการกดยืนยัน (Confirm) เพื่อส่งเรื่องเข้าสู่กระบวนการจัดตารางเรียนต่อไป

**💡 รูปแบบการคืนค่า (Return Status):**

API ในชุดนี้มีการออกแบบ Error Handling ที่อิงตาม HTTP Status Code เป็นหลัก ซึ่งคนรับงานต่อต้องทราบ ได้แก่:

- `200` = สำเร็จ (Success)
- `400` = ข้อมูล/Parameter ไม่ครบถ้วน (Bad Request)
- `403` = ติดเงื่อนไข Business Logic (เช่น หน่วยกิตเกิน, ตารางสอบชน)
- `500` / `507` = ระบบขัดข้อง / Exception (Internal Error)

## ⚙️ 2. สรุปฟังก์ชันการทำงานหลัก (Core Procedures)

ผมได้จัดกลุ่ม Procedures ทั้งหมดออกเป็น 4 กลุ่ม เพื่อให้ง่ายต่อการทำความเข้าใจดังนี้ครับ:

### กลุ่มที่ 2.1: การจัดการข้อมูล (CRUD Operations)

เป็นกลุ่มที่รับ Action จากนักศึกษาเมื่อกดปุ่มบนหน้าจอเว็บ

- **`PRE_REG_INS` (เพิ่มรายวิชา):** ฟังก์ชันหลักตอนนักศึกษากด Add วิชา
    
    - _Business Logic 🚨:_ ระบบจะทำการเช็ค **"ตารางสอบซ้ำซ้อน"** หากพบว่าชนกัน (และไม่ใช่เด็กปี 4) ระบบจะดีด Error 403 ทันที นอกจากนี้ยังเช็คสถานะวิชาว่าเป็นช่วงเปิดรับหรือไม่
        
- **`PRE_REG_DEL` (ลบรายวิชา):** ฟังก์ชันตอนนักศึกษากดลบวิชาที่เลือกไว้
    
    - _Business Logic 🚨:_ ความซับซ้อนอยู่ที่ **การเช็คขั้นต่ำ (Credit Min)** หากนักศึกษากดยืนยันไปแล้ว ระบบจะไม่ยอมให้ลบรายวิชาออกถ้ายอดหน่วยกิตรวมจะลดลงต่ำกว่าเกณฑ์ขั้นต่ำที่มหาวิทยาลัยกำหนด (เว้นแต่เป็นช่วงลงทะเบียนล่าช้า Late Period)
        
- **`PRE_REG_UPD` (แก้ไขหมายเหตุ):** ใช้สำหรับอัปเดตช่อง Remark (เหตุผลที่ขอลง)
    
- **`PRE_REG_INS_CHANGE_SECTION` (เปลี่ยนกลุ่มเรียน):** ย้าย Section โดยระบบจะทำการลบข้อมูล Section เก่า และ Insert Section ใหม่ให้แบบ Auto
    

### กลุ่มที่ 2.2: การยืนยันสิทธิ์ (Confirmation & Validation)

- **`PRE_REG_CONFIRM` (ยืนยันการแจ้งความประสงค์):** ฟังก์ชันที่สำคัญที่สุด! ถูกเรียกเมื่อนักศึกษากดปุ่ม "ยืนยัน"
    
    - _Business Logic 🚨:_ ระบบจะดึง **"หน่วยกิตสูงสุด (Credit Max)"** ของนักศึกษาคนนั้นมาคำนวณ หากวิชาที่ขอไปทั้งหมดมีหน่วยกิตรวมกัน "เกิน" กว่าเกณฑ์ ระบบจะบล็อกและดีด Error แจ้งว่าหน่วยกิตเกิน นอกจากนี้ยังมีการแยก Flow การทำงานระหว่างช่วงเวลาปกติ (Period 1) และช่วงล่าช้า (Period 2, 3) ออกจากกันชัดเจน
        

### กลุ่มที่ 2.3: การดึงข้อมูลแสดงผล (Data Fetching)

- **`PRE_REG_SEL`:** ดึงตารางรายวิชาที่นักศึกษาคนนั้นกดเลือกไว้ เพื่อมาวาดบนหน้าจอเว็บ (แยกตามช่วงเวลาปกติและล่าช้า)
    
- **`PRE_REG_CREDIT_SEL`:** ดึง "ยอดสรุปหน่วยกิต" และ "จำนวนรายวิชา" (แยกยอดกดยืนยันแล้ว กับยอดที่ยังไม่ยืนยัน)
    
- **`PRE_REG_APPROVE_RESULT`:** เช็คสถานะว่ารายวิชาที่ยืนยันไป "ผ่านการอนุมัติ" หรือยัง (ใช้ `APPROVE_FLAG`)
    
- **`PRE_REG_SEL_INSTRUCTOR`:** ดึงชื่ออาจารย์ผู้สอน เพื่อให้นักศึกษาทราบว่าต้องไปติดต่อใครเพื่อขออนุมัติเป็นกรณีพิเศษ
    

### กลุ่มที่ 2.4: สำหรับเจ้าหน้าที่และแอดมิน (Staff / Admin Actions)

- **`REQ_CONFIRM_CLASS_ADJUST_UPD` (ปรับจำนวนที่นั่งรับ):** เจ้าหน้าที่สามารถเพิ่ม/ลด จำนวนที่นั่ง (`TOTALSUMSEAT`) ได้ โดยระบบจะทำการบันทึกประวัติการแก้ไขลงในตาราง **`CLASS_ADJUST_LOG`** พร้อมเก็บเลข IP Address และรหัสพนักงานไว้ตรวจสอบย้อนหลัง
    
- **`REQ_CONFIRM_EXTRA_UPD`:** เจ้าหน้าที่ใช้อัปเดตสถานะ "อนุญาตพิเศษ" (`Extra_Register`)
    
- **`REQ_CONFIRM_SEL` & `REQ_CONFIRM_DTL_SEL`:** ชุดดึงข้อมูลสถิติยอดคนขอที่นั่ง, ยอดคนยืนยัน, และที่นั่งว่าง สำหรับหน้าจอฝั่งเจ้าหน้าที่หรืออาจารย์ผู้ประสานงาน


# System Documentation

## Procedure : MFU_API.MAINREGISTER_PREREG

## 1. Overview

`MFU_API.MAINREGISTER_PREREG` เป็น Oracle PL/SQL Stored Procedure
ที่ใช้สำหรับตรวจสอบและประมวลผลการแจ้งความประสงค์ลงทะเบียนเรียนล่วงหน้า
(Pre‑Registration)

หน้าที่หลักของ Procedure นี้คือ

-   ตรวจสอบสถานะนักศึกษา
-   ตรวจสอบเงื่อนไขการลงทะเบียน
-   ตรวจสอบหน่วยกิต
-   ตรวจสอบตารางเรียนชนกัน
-   ตรวจสอบตารางสอบชนกัน
-   ตรวจสอบช่วงเวลาการลงทะเบียน
-   ตรวจสอบเงื่อนไขหลักสูตร

หากพบข้อผิดพลาด ระบบจะส่งค่ากลับผ่าน

-   `VERRID`
-   `VERRSTR`

เพื่อแจ้ง Error ให้กับระบบ Frontend

# 6. Flow การทำงานของ Procedure

## Step 1 ตรวจสอบนักศึกษา

ตรวจสอบว่ามีรหัสนักศึกษาในระบบหรือไม่
หากไม่พบ

Error 1001 -- Student not found

อ่านข้อมูลจาก
  
STUDENTMASTER
STUDENTSTATUS

หากสถานะไม่อนุญาตให้ลงทะเบียน
Error 1002 -- Student status invalid

## Step 3 ตรวจสอบ Pre‑Register Header
อ่านข้อมูลจาก

-   PROGRAM
-   STUDENTSTATUS

------------------------------------------------------------------------
## Step 5 ตรวจสอบ Lock การลงทะเบียน

Error 1010
